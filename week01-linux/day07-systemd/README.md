# Day 07 – 서비스 관리(systemd) 개념 이해

Day 7에서는 서버 환경에서
프로세스를 직접 실행하는 방식과
운영체제가 서비스를 관리하는 방식의 차이를 이해하는 데 집중했다.

명령어 사용보다는
“왜 systemd가 필요한가”와
“서버는 어떻게 안정적으로 서비스 상태를 유지하는가”를
개념적으로 정리하는 것이 목표였다.

---

## systemd란 무엇인가

systemd는 리눅스 운영체제의 서비스 관리자(init system)이다.

개별 프로세스를 사람이 직접 실행하고 관리하는 대신,
운영체제가 서비스 단위로 생명주기를 관리하도록 한다.

systemd는 다음과 같은 역할을 담당한다.

- 서비스 시작 / 종료
- 서비스 상태 관리
- 비정상 종료 시 재시작
- 시스템 부팅 시 자동 실행
- 로그 수집 및 관리

---

## 왜 systemd가 필요한가

직접 프로세스를 실행하는 방식에는 한계가 있다.

- 터미널을 종료하면 프로세스가 함께 종료될 수 있다
- 서버 재부팅 시 서비스가 자동으로 올라오지 않는다
- 장애 발생 시 상태를 일관되게 관리하기 어렵다

systemd는 이러한 문제를 해결하기 위해
서비스를 운영체제 차원에서 관리하도록 설계되었다.

---

## 서비스 단위(unit) 개념

systemd는 서비스를 프로세스가 아닌
**unit**이라는 단위로 관리한다.

그중 가장 대표적인 unit이 `service` unit이다.

service unit은 다음과 같은 정보를 포함한다.

- 어떤 명령을 실행할지
- 언제 실행할지
- 실패 시 어떻게 처리할지
- 부팅 시 자동 실행 여부

---

## service unit 파일 구조 이해

일반적인 service unit 파일은 다음과 같은 구조를 가진다.

```ini
[Unit]
Description=My Application Service

[Service]
ExecStart=/path/to/app.sh
Restart=always

[Install]
WantedBy=multi-user.target
[Unit] : 서비스의 설명과 의존성

[Service] : 실제 실행 명령과 동작 방식

[Install] : 부팅 시 서비스 활성화 여부
