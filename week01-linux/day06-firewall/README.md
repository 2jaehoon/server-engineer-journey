# Day 6 – Firewall 개념과 접근 흐름

Day 6에서는 방화벽이 서버 아키텍처에서
어디에 위치하며 어떤 역할을 하는지를
접속 흐름 관점에서 정리했다.

명령어 사용 자체보다,
“서버는 정상인데 접속이 안 되는 이유”를
논리적으로 추적하는 것이 목표였다.

---

## 방화벽의 위치와 역할

방화벽은 프로세스나 포트보다 바깥에서 동작한다.
외부에서 들어오는 네트워크 요청은
서버의 프로세스에 도달하기 전에
먼저 방화벽 정책을 통과해야 한다.

요청 흐름은 다음과 같다.

외부 요청  
→ 방화벽  
→ 포트 / 소켓  
→ 프로세스

따라서 프로세스와 포트가 정상이어도,
방화벽에서 차단되면 서버는 응답할 수 없다.

---

## WSL 환경에서의 제약

현재 학습 환경은 WSL(Windows Subsystem for Linux)로,
네트워크 방화벽은 리눅스가 아닌
Windows 쪽에서 관리된다.

이로 인해 UFW와 같은 리눅스 방화벽 도구를
직접 설치하거나 실습하는 데에는 제약이 있었다.

하지만 이는 방화벽 개념을 이해하는 데
문제가 되지 않으며,
실제 서버 환경(Ubuntu)에서는
UFW 또는 iptables를 통해
인바운드 트래픽을 제어하게 된다.

---

## 접속 장애 점검 흐름 정리

서버 접속이 되지 않을 때는
다음 순서로 점검하는 것이 효율적이다.

1. 프로세스가 실행 중인지 확인한다 (ps)
2. 해당 프로세스가 포트를 LISTEN 중인지 확인한다 (ss)
3. 실제 연결(ESTABLISHED)이 발생하는지 확인한다
4. 마지막 단계에서 방화벽 규칙을 점검한다

방화벽은 항상 마지막에 확인해야 할 요소이며,
대부분의 문제는 그 이전 단계에서 해결된다.

---

## 느낀 점

- 방화벽은 막연한 네트워크 설정이 아니라,
  서버 접근 흐름의 일부라는 것을 이해하게 되었다
- 접속 문제를 네트워크부터 의심하기보다,
  서버 내부 상태부터 차근차근 확인하는 것이 중요하다고 느꼈다

---

## 한 줄 정리

방화벽은 서버의 가장 바깥에서 동작하며,
프로세스와 포트가 정상이어도
요청을 차단할 수 있는 마지막 관문이다.
